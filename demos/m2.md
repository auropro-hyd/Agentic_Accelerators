## Demo flow (10 minutes)

### Step 1 — Where we left off (1 min)

> "Last week M1 produced this bundle from the source schema. M2 doesn't
> change anything in the schema/ tree — it adds profiles/ and readiness/.
> No regressions to M1 work."

```bash
ls bundle/schema/tables/ | wc -l            # 15 (12 original + 3 quality_*)
ls bundle/                                  # source.*, schema/, bundle.json
```

### Step 2 — Profile every column (3 min)

```bash
uv run dla profile --config config/examples/postgres_minimal.yaml
```

Expected:

```
Profiling complete.
  source_id:       fixture_postgres
  mode:            sampling
  profiles:        83
  errors:          1
```

> "Sampling mode by default — up to 10k rows per column, configurable in
> the YAML. One column couldn't be profiled — that's the deliberately empty
> table. We'll see it surface in readiness as a critical issue."

Open one profile:

```bash
cat bundle/profiles/public.quality_users.referral_code.md
```

Read out the frontmatter — `null_rate: 0.7`, `sample_size: 10`,
`distinct_count: 3`, `top_values` showing what's in the data. **This is
the grounding signal M3 will feed the LLM.**

Open the all-null one:

```bash
cat bundle/profiles/public.quality_users.middle_name.json | head -20
```

`null_rate: 1.0` everywhere. The column is in the schema but unused in
practice.

### Step 3 — Readiness report (3 min)

```bash
uv run dla readiness --config config/examples/postgres_minimal.yaml
```

Expected output (numbers may vary by a few units run-to-run depending on
sampling order, but the headline shape is stable):

```
Readiness report complete.
  source_id:    fixture_postgres
  total issues: 47
    critical  7
    warning   1
    info      39
  by type:
    all_null_column   5
    broken_fk         1
    constant_column   39
    empty_table       1
    high_null_rate    1
```

> "Seven critical issues are the headline:
> - one **broken FK** — `quality_invoices.customer_id` has values that
>   don't exist in `customers.id`,
> - one **empty table** — `quality_empty_orders` has zero rows,
> - five **all-null columns**, including the obviously-unused `middle_name`."

Open the human-readable summary:

```bash
cat bundle/readiness/readiness.md
```

The critical block lists each issue with its artifact id and the affected
artifact references. Open one issue in detail:

```bash
cat bundle/readiness/issues/readiness_issue.broken_fk.0001.md
```

> "Every issue carries a remediation suggestion in plain language. The SME
> in the M4 web UI will be able to walk this list and either fix or accept
> each. Today, this is the input to the conversation."

### Step 4 — Why the noise? (1 min)

> "39 `constant_column` issues — most of those are tiny seed tables where
> we genuinely only have 1-3 rows; with such a small sample distinct=1 is
> arithmetically inevitable. Severity is `info`, not warning, so they're
> below the fold for any SME walking the report.
>
> In a real engagement with thousands of rows, `constant_column` becomes
> meaningful — a literal never-used flag column."

This is one of the course-correction questions for today: do we want a
minimum-sample-size threshold before flagging `constant_column`?

### Step 5 — Idempotency, again (1 min)

> "M1 invariant carries to M2: re-running profile + readiness on an
> unchanged source produces zero diffs."

```bash
find bundle -type f -exec stat -f "%m %N" {} \; | sort > /tmp/before.txt
wc -l /tmp/before.txt   # ~496 files

uv run dla profile --config config/examples/postgres_minimal.yaml
uv run dla readiness --config config/examples/postgres_minimal.yaml

find bundle -type f -exec stat -f "%m %N" {} \; | sort > /tmp/after.txt
diff /tmp/before.txt /tmp/after.txt   # zero lines = identical
```

### Step 6 — Course-correction questions (1 min)

(Also captured in `demos/m2-course-correction.md`.)

1. Are the severity thresholds reasonable? `null_rate ≥ 0.5` → warning;
   `≥ 0.9` → critical. The seeded `referral_code` (~70% null) lands at
   warning today.
2. Should `unprofiled` be `info` or escalated to `critical`? An unprofiled
   column blocks M3 description generation.
3. Are the profile sample sizes large enough? Default 10k rows; demo used
   10 rows because the fixture is small.
4. Should `constant_column` require a minimum sample size before firing
   (e.g., 50 rows)? Today it fires on 1-3-row tables.
