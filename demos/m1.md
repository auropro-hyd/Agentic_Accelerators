## Demo flow (10 minutes)

### Step 1 — The problem (1 min)

> "Every L1 engagement starts the same way: someone points us at a client database, hands us a couple of dictionaries and a dbt manifest, and asks for a knowledge bundle the agents can consume. M1 is the first piece of that workflow — pointing the accelerator at a source and producing a confidence-tagged bundle of the schema. No AI yet, no SME UI yet — just the deterministic foundation everything downstream builds on."

Open `docs/Accelerators_Scope_v5.md`, scroll to the L1 section, point at the highlighted goal.

### Step 2 — Postgres discovery (3 min)

Open a clean terminal and run:

```bash
uv run dla discover --config config/examples/postgres_minimal.yaml
```

Expected output:

```
Discovery complete.
  source_id:       fixture_postgres
  bundle:          .../bundle
  tables:          12
  columns:         72
  relationships:   10
  indexes:         4
```

Then walk through the bundle:

```bash
tree -L 3 bundle/
```

Highlight:

- `bundle/source.{md,json}` — the configured source
- `bundle/schema/tables/public.orders.md` — every table as a paired markdown + JSON file
- `bundle/schema/columns/public.orders.status.md` — column with `data_type`, `normalized_type`, nullability
- `bundle/schema/relationships/public.orders.customer_id->public.customers.id.json` — the declared FK, tagged `confidence: Explicit`, `signals: [declared_fk]`
- `bundle/bundle.json` — top-level manifest with `schema_version: 1.0.0-m1`

Open one column file in the IDE:

```bash
cat bundle/schema/columns/public.orders.status.md
```

Read out the frontmatter: `artifact_id`, `provenance: discovered`, `data_type: varchar(32)`, `is_nullable: false`. **This is what the SME would edit in M3+ when descriptions land.**

### Step 3 — CSV provider, same bundle shape (2 min)

> "Same engine, different provider. CSV folder connector uses pandas type inference instead of SQLAlchemy reflection — but the output is the same bundle shape, so anything M2+ writes (profiles, descriptions) drops in without restructuring."

```bash
uv run dla discover --config config/examples/csv_folder.yaml
ls bundle_csv/schema/
```

Open the inferred relationship — this is the interesting one:

```bash
cat "bundle_csv/schema/relationships/orders.customer_id->customers.id.md"
```

Highlight:

- `relationship_type: inferred_fk` (no declared FK in CSV)
- `confidence: Strong`
- `signals: [name_match, type_match, value_overlap]` — three independent signals agreed; that's why the tagger lifted it to Strong, not Weak

> "If only the name had matched, this would have been Weak. The tagger requires either two signals or explicit value-overlap to call it Strong."

### Step 4 — Idempotency (2 min)

> "Critical M1 invariant: re-running discover against an unchanged source produces zero diffs. This is the foundation for M3's SME-edit preservation — if discover wrote noise every run, no SME edit would ever be safe."

```bash
# Snapshot mtimes before
find bundle -type f -exec stat -f "%m %N" {} \; | sort > /tmp/before.txt
wc -l /tmp/before.txt    # ~199 files

# Re-run discover
uv run dla discover --config config/examples/postgres_minimal.yaml

# Compare
find bundle -type f -exec stat -f "%m %N" {} \; | sort > /tmp/after.txt
diff /tmp/before.txt /tmp/after.txt
echo "exit=$?"           # 0 = identical
```

Expected: `diff` produces no output. Every file untouched.

### Step 5 — Failure modes (1 min)

```bash
# Bad password -> exit code 2 (connection)
DLA_DB_PASSWORD=wrong uv run dla discover --config config/examples/postgres_minimal.yaml
echo "exit=$?"   # 2

# Missing config -> exit code 3 (config)
uv run dla discover --config /tmp/missing.yaml
echo "exit=$?"   # 3
```

> "The CLI exits with documented codes so a CI gate or wrapper script can react. Connection failures are recoverable (retry); config errors aren't (operator fix required)."
